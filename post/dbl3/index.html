Database Implementation Lecture3 Notes

Lecture 3

parity disk: - def of parity bit - detect bit flip, number of bits that are corrupted - eg. 4 data blocks and a parity block

B1 B2 B3 B4 P, each on different disks D1 D2 D3 D4 D5 - how to calculate the parity block? - P = B1 XOR B2 XOR B3 XOR B4 - Read: - if B2 becomes unavailable: - read(B2) becomes read(B1)(B2)(B3)(P) - B2 = P XOR B1 XOR B3 XOR B4 - read factored by 4 since now to read B2 has to read 4 blocks instead of read 4 directly - on recovery, do the same computation in the background and write to the new disk - may take a while, have to scan data on all disks - what happen to write: - have to involve the parity disk - W(B2) - update B2 block - update P: P = newB2 XOR oldB2 XOR oldP (have to know old parity) - After failure of D2, what happen when writing to B2? 1. recover oldB2 = B1 XOR B3 XOR B4 XOR P 2. don’t have to recover, just find newP = B1 XOR B3 XOR B4 XOR newB2

  • New Config:

    D1 D2 D3 D4 D5

    B1 B2 B3 B4 P1-4

    B5 B6 B7 B8 P5-8

    • not good since both parity disks on D5, every write has to touch D5 thus limited by the throughput of D5
    • Solution:
    • Alternating the parity disks: B1 B2 B3 B4 P1-4

    B5 B6 B7 P5-8 B8

    • RAID5: most popular configuration, don’t have a big overhead, has resilience to one failure

      • overhead: 4 disks of data and 1 disk of parity, thus 25% of overhead compare to 100% overhead when doing Mirroring what doubles the data

    • RAID6: more disks for parity disk, resilience to 2 disk failures

  • Use RAM to store some of the data

    • for this purpose, create Buffer pool
    • Buffer Pool
    • an array of memory-resident blocks populated on demand from resistant storage
    • RAM-resident BP
    • Say 8GB of RAM, 4KB data blocks -> 2*10^6 pages in BP

    • read

      • what if out of blocks?
      • Least Recently Used
        • replace blocks that are least recently Used
        • pitfall: sequential flooding:
        • in a sequential scan, read pages in a 10GB table one by one -> every time read a block, it becomes the most recent but never goes back to it. Thus evicting all the others, which can be shared to others
        • Solution
          1. BP quota: per user/query/table
          2. Bring in a new page specifically for sequential scan and mark it with hate, keep replacing those hated pages while running sequential scan thus touching very few other pages

      • Least frequently used

        • pitfall: transient workload: may use some data frequently for a period of time and then never use them bring some count very high
        • solution: Aging:
        • in a certain amount of time, decrement all counts, let them eventually lose their dominance
        • could be multiplicative (halving) or additive (decrementing)
        • why is it not so expensive: only do once a while

      • write

      • mark a page as “dirty”

        • but not every write to BP write to disk at the same time.
        • on evicting, if it’s dirty, write to disk.
        • Force writing to disks:
        • Cons: if immediately also writing to disk, two much overhead, takes 2ms for every write
        • No force:
        • to be resilient to failure: log, to have a recovery algorithm
        • Steal:
        • to be resilient to aborts: log
        • can accumulate multiple writes together

    • What’s on a page:

      • assume data records are stored contiguously
      • if have indexes that point to records:
      • storing Block: ID + byte address
      • problem?
        • wants to defragmentation of blocks (changing physical address), but if there are some references to the byte address, then cannot move the records
      • instead of using Byte address, use Block: ID + offset in slot array (RID), in the slot array there are pointers to the records’ byte address but now those byte address can change to defragmentation to make data blocks contiguous
