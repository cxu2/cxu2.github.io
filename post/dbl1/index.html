DB Impl Lecture 1

Lecture 1

  • Consistency

    • integrity constraints – don’t let transactions commit
    • concurrency
      • conflicting writes
      • locks
      • critical sections
      • semaphores
      • mutexes
    • concurrency control protocols
      • make sure half-completed transactions/money loss won’t happen

  • atomicity

    • either success or all abort

  • Isolation

  • Durability & fault-tolerance

    • machine goes down, if transactions are committed, on reboot the data written should be there
    • save in ram / disk
    • recovery
    • disk failure
    • network failure for distributed databases, leading to partitioning
    • RAID – have a redundant set of disks and being able to sub on
    • replications to avoid natural disasters – duplicated data centers
    • roll back – so have to do extra overhead to record before-values

  • Security & data integrity

  • Compiler for transactions

    • into a set of function calls

  • Query optimizer

    • orders of operations especially with multiple joins
    • optimal performance

  • Index

    • Hashtable / B-tree
    • increase efficiency for lookup

  • Materialized View ?

    • create View statement
    • actually take the view and explicitly keep the date and store it
    • down sides:
      • storing data redundantly and takes up more space
      • data in the Materialized view might be stale – the underlying data might have updated
      • fix: to the Materialized view incrementally instead of always from scratch
